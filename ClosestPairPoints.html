1 — The Problem & Why It Matters

You have a list of locations on a map — cities, drones, atoms in a simulation, players in a multiplayer game — and you need to answer one question:

"Of all these points, which two are closest together?"

It sounds simple, and for a handful of points it is. The interesting bit is doing it fast when there are millions of points — which is exactly the regime where it matters most in practice.

Where this comes up

  • Collision detection — physics engines and games ask "are any two objects about to bump?" every frame.
  • Air-traffic control — the system that warns "two aircraft on a near-collision course."
  • Geographic search — "find the nearest hospital / store / gas station."
  • Bioinformatics & clustering — "which two DNA samples (encoded as vectors) are most similar?"
  • Computer vision — matching feature points between two photographs.

The formal problem

Given n points P_1, P_2, …, P_n in the plane, find the pair (P_i, P_j) with minimum Euclidean distance:

dist(P_i, P_j) = √( (x_i − x_j)² + (y_i − y_j)² )
Goal

Solve this in O(n log n) — much faster than the naive O(n²) of comparing every possible pair. At n = 10⁶ that's the difference between ~5·10¹¹ and ~2·10⁷ operations: a 25 000× speed-up.

2 — Brute-Force Baseline

The most direct algorithm: compute every pair's distance and remember the smallest:

best = ∞ for i = 0 .. n−1: for j = i+1 .. n−1: d = dist(P[i], P[j]) if d < best: best = d, bestPair = (i, j) return bestPair

This makes n(n−1)/2 ≈ n²/2 distance computations — O(n²).

We need to avoid examining every pair. The key observation: most pairs are obviously far apart and we shouldn't waste time on them. The challenge is recognizing that fact without inspecting each pair individually — which is exactly what divide-and-conquer accomplishes.

3 — The Big Idea: Divide and Conquer

Imagine drawing a vertical line that splits the points into a left half and a right half. The closest pair in the entire set must be one of three things:

  • both points in the left half
  • both points in the right half
  • one in each half (a "cross-boundary" pair)

The first two cases are smaller versions of the same problem — perfect for recursion. The cross-boundary case is what makes the algorithm interesting.

The crucial leverage

After recursing, we already know the best distance δ from each half. Any cross-pair worth caring about must beat δ. Both points of such a pair must therefore lie within horizontal distance δ of the split line — otherwise their x-gap alone is already more than δ. So we only need to search a thin strip of width 2δ around the splitter.

The full plan, in one breath

  • Divide. Split the x-sorted points in half at the median x.
  • Conquer. Recursively find the closest pair in each half. Take δ = min(δ_L, δ_R).
  • Combine. Scan the strip of width 2δ for any cross-pair beating δ. Return the overall best.

4 — Divide

Before the first recursive call, sort all points by x once. Maintain a y-sorted view as well — both views can be carried through the recursion in O(n) per level instead of re-sorting.

Splitting

At each call, take the median index. The x-coordinate at that index defines the splitter line. Everything to the left becomes L, everything to the right becomes R:

mid = ⌊|P| / 2⌋ splitX = P_x[mid].x L = P_x[0..mid−1] (n/2 points) R = P_x[mid..n−1] (n/2 points)

Recurse on each half. They each return their own closest-pair distance: δ_L and δ_R.

Base case. Once a sub-problem has 3 or fewer points, just brute-force it. With ≤ 3 points only ≤ 3 pairs exist, and brute force avoids further recursion overhead.

5 — The Strip (Conquer Step)

After both halves return, set δ = min(δ_L, δ_R). Now we hunt for any cross-pair that beats δ.

Building the strip

Collect every point within horizontal distance δ of the splitter:

strip = { p ∈ P : | p.x − splitX | < δ }

Sort the strip by y-coordinate (or extract from the maintained y-sorted view).

Why this is enough

Suppose a cross-pair (p, q) has dist(p, q) < δ. Their horizontal gap satisfies |p.x − q.x| ≤ dist(p, q) < δ. Since one is in L (so p.x ≤ splitX) and the other is in R (so q.x ≥ splitX), both must sit within distance δ of the splitter. So any cross-pair we care about lives entirely inside the strip.

What if the strip is large?

The strip can hold up to n points in pathological cases. Naively scanning all pairs in the strip is O(strip²), which would wreck the algorithm. The 7-Point Lemma rescues us.

6 — The 7-Point Lemma (Why the Strip Scan is Linear)

Walk the strip in increasing y. For each point p, only compare it against the next ≤ 7 points in the y-order. That's it. The whole strip scan is O(strip size).

Geometric proof sketch. Inside the strip, no two points from the same side can be closer than δ to each other (otherwise that half's recursion would already have found a smaller δ). Now look at the 2δ × δ rectangle above any point p. Tile that rectangle with δ/2 × δ/2 sub-cells — that's a 4 × 2 = 8-cell grid. Each sub-cell can hold at most one point (any two points in a single δ/2 × δ/2 cell would be < δ apart, but they're forced to be ≥ δ apart). So the 2δ × δ window contains at most 8 points, including p itself — hence at most 7 candidates to compare against.

The scan in code

for i = 0 .. |strip|−1: for j = i+1 .. min(i+7, |strip|−1): if strip[j].y − strip[i].y ≥ δ: break // can't improve any further d = dist(strip[i], strip[j]) if d < δ: δ = d bestPair = (strip[i], strip[j])

The inner loop is bounded by a constant (7), so the entire strip scan is O(|strip|) ≤ O(n).

7 — Recurrence & Why It Gives O(n log n)

Combining the three steps: each call solves two sub-problems of size n/2, then does O(n) linear strip work to combine.

T(n) = 2 · T(n/2) + O(n) ⇒ T(n) = O(n log n)

This is exactly the merge-sort recurrence (Master Theorem, case 2). The recursion tree has depth log₂ n; each level performs O(n) total work; multiplying gives O(n log n).

The initial sort is also O(n log n). By maintaining a y-sorted view alongside the x-sorted view (filter into left/right at each level in O(n)), no further sorting is needed inside the recursion — that's how the strip step stays linear instead of O(n log n) per level.

The same algorithmic shape (divide on the median, combine in a thin strip) extends to higher dimensions, with slightly larger constants in the lemma — every fast geometric query you encounter borrows from this template.
