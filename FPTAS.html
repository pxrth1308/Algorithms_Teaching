1 — The Problem & Why It Matters

You're packing for a flight with a strict weight limit. In front of you sit several items. Each one has a value (how much you'd enjoy taking it) and a weight. You can either pack an item entirely or leave it behind — no half-bottles, no half-jackets. The question:

"Which subset of items should I pack to maximize total value without exceeding the weight limit?"

This is the 0/1 Knapsack Problem, one of the most famous combinatorial optimization problems in computer science.

Where it shows up

  • Capital budgeting — pick which projects to fund under a fixed budget to maximize ROI.
  • Cargo loading — what to load onto a ship/plane/truck under a weight or volume limit.
  • Cloud resource allocation — schedule jobs onto a server with limited CPU/RAM, maximize utility.
  • Cutting stock & manufacturing — choose pieces to cut from a roll of material.
  • Cryptography — early public-key schemes (Merkle-Hellman) were knapsack-based.

The formal problem

Given n items with values v_1, …, v_n and weights w_1, …, w_n, and a knapsack capacity W, choose a subset S ⊆ {1..n} with ∑i∈S w_i ≤ W that maximizes ∑i∈S v_i.

Goal of an FPTAS

0/1 Knapsack is NP-hard, so we don't expect a polynomial-time exact algorithm. An FPTAS (Fully Polynomial-Time Approximation Scheme) accepts an error tolerance ε > 0 chosen by the user and returns a solution with value ≥ (1 − ε) · OPT, in time polynomial in both n and 1/ε. Pick ε = 0.01 for "within 1 % of optimum"; pick ε = 0.0001 for tighter; runtime grows gracefully.

2 — Why Knapsack Is Hard: NP-Hardness & Pseudo-polynomial DP

0/1 Knapsack is one of Karp's original 21 NP-complete problems (1972). Unless P = NP, no algorithm runs in time polynomial in the input length and produces the exact optimum on every instance.

Brute force

Try every subset: 2ⁿ possibilities. Fine for n = 25, hopeless for n = 60.

Two natural DPs

  • Capacity-indexed: dp[i][w] = max value using items 1..i with total weight ≤ w. Time / space: O(n · W).
  • Value-indexed: dp[i][V] = min weight to reach value V using items 1..i. Time / space: O(n · V_total) where V_total = ∑ v_i.

Why these DPs are called "pseudo-polynomial". The runtime is polynomial in the numerical magnitude W or V_total, not in the length log W of the input. With genuine inputs like v_i = 10⁹, V_total can be 10¹¹ — a table that won't fit in memory. Truly polynomial means polynomial in the bit-length of the numbers, which neither DP achieves.

So we have a dilemma: brute force is exponential in n; DP is exponential in the size of the value/weight numbers. The way out is to not insist on exact answers.

3 — The Idea: Approximate, Don't Solve Exactly

NP-hardness blocks exact polynomial-time solutions, but it leaves a back door wide open: solutions that are nearly optimal. For knapsack we can get within any user-chosen tolerance ε of OPT, in genuinely polynomial time. The trick is built in two layers.

Why exact value-indexed DP fails

The value-indexed DP is fast when V_total is small. With huge values it has too many columns. So the question is: can we shrink the values without changing the answer too much?

The plan

  • Scale every value v_i down by a chosen factor K, then floor: v'_i = ⌊v_i / K⌋.
  • Run the value-indexed DP on the scaled values. The table now has only O(n²/ε) columns — polynomial in everything.
  • Argue that flooring loses at most K per item. Add up: total error is at most n·K = ε·v_max ≤ ε·OPT. So we still get (1 − ε) · OPT.

That's the entire FPTAS in three bullets. The next four sections fill in the details.

4 — The Scaling Trick

Pick the scaling factor:

K = ε · v_max / n where v_max = max v_i

Replace each value:

v'_i = ⌊ v_i / K ⌋

Why this exact K?

Two competing forces:

  • Big K → values shrink a lot → DP table is tiny → fast, but rounding error is large.
  • Small K → values barely change → table is big → slow, but rounding error is tiny.

The choice K = ε·v_max / n is the sweet spot: each item loses at most K when floored, total loss ≤ n·K = ε·v_max ≤ ε·OPT — exactly the bound we want — while each scaled value is bounded by v_max / K = n/ε, so V_total ≤ n · (n/ε) = n²/ε.

Why the table is now polynomial

V_total ≤ n²/ε depends only on n and 1/ε — not on the magnitude of the original values. A 64-bit value worth a billion is now scaled to a small integer, and the DP runs in time bounded purely in the input length.

5 — Scaled DP (with the Exact DP for Comparison)

Run the value-indexed DP on the scaled values v'_i:

dp[i][V] = min weight to reach scaled value V using items 1..i
  • Base: dp[0][0] = 0,  dp[0][V>0] = ∞  (can't reach positive value with zero items).
  • Recurrence: dp[i][V] = min( dp[i−1][V],  dp[i−1][V − v'_i] + w_i ). The two choices: skip item i (left) or take it if it fits (right).
  • Answer: the largest V with dp[n][V] ≤ W. Recover the chosen items by tracing the recurrence backward.

Side-by-side with the exact DP

The visualizer below fills two tables in lockstep, item by item:

  • Left (exact): dp[i][w] = max value with capacity ≤ w. Has W + 1 columns — width depends on the input number W.
  • Right (FPTAS): dp[i][V] on scaled values. Has at most n²/ε + 1 columns — width depends only on n and 1/ε.

Watch how the two tables grow: the exact one balloons with W, while the FPTAS one stays bounded no matter how huge the original values are.

6 — Why the Answer Is Within (1 − ε) of OPT

Let S* be the truly optimal subset (we don't see it, but it exists). For each item i ∈ S*:

v_i ≥ K · v'_i ≥ v_i − K

(The first inequality is "scale, then unscale"; the second is the floor losing at most K.) Summing over the items in S*:

∑i∈S* v_i − K · ∑i∈S* v'_i ≤ |S*| · K ≤ n · K = ε · v_max ≤ ε · OPT

Since the FPTAS DP finds the subset that maximizes ∑ v'_i (under the same weight constraint), it picks something at least as good as S* in scaled-value terms. Translating back to true values, the recovered solution has value at least:

OPT − ε · OPT = (1 − ε) · OPT
The trade-off knob

Pick ε = 0.5: half-OPT guarantee, very fast (~2n³). Pick ε = 0.01: 99 % guarantee, 50× more work. Pick ε = 0.001: 99.9 %, 500× more work. You set the dial; the algorithm responds polynomially.

7 — PTAS vs FPTAS (a small but important distinction)

Both schemes deliver (1 − ε)-approximations. The difference is how the runtime depends on ε:

SchemeRuntimeBehaviour as ε → 0
PTASpoly(n) for any fixed ε1/ε can sit in the exponent, e.g. O(n1/ε) — explodes for tiny ε
FPTASpoly(n, 1/ε)strictly stronger; tiny ε only adds polynomial cost

Knapsack admits an FPTAS (this lesson). Bin Packing and Euclidean TSP admit only a PTAS. Vertex Cover, MAX-3-SAT, and (assuming the Unique Games Conjecture) many others admit neither — they have hard approximation lower bounds.

8 — Complexity

MethodTimeNotes
Brute force (try all subsets)O(2ⁿ)Exact, exponential
Pseudo-polynomial DPO(n · V_total)Exact, but scales with the magnitude of values
FPTASO(n³ / ε)(1 − ε)-approximation, polynomial in n and 1/ε

Sanity check at n = 100, ε = 0.01

  • Brute force: 2¹⁰⁰ ≈ 10³⁰ — never finishes.
  • Pseudo-poly DP with v_i ~ 10⁹: ~10¹¹ ops — minutes to hours.
  • FPTAS: n³/ε = 10⁸ ops — a fraction of a second, with answer guaranteed within 1 % of optimum.

9 — Worked Example

Items (value v, weight w): (6, 4), (8, 5), (5, 3) Capacity W = 10, ε = 0.2, n = 3, v_max = 8 Step 1 — choose K K = ε · v_max / n = 0.2 · 8 / 3 ≈ 0.533 Step 2 — scale values v'_i = ⌊ v_i / K ⌋ v'_1 = ⌊6 / 0.533⌋ = 11 v'_2 = ⌊8 / 0.533⌋ = 15 v'_3 = ⌊5 / 0.533⌋ = 9 Step 3 — value-indexed DP on scaled values V_total ≤ 11 + 15 + 9 = 35 (table has 36 columns) Best scaled value with weight ≤ 10: items 1 + 2 → V' = 26, weight = 4 + 5 = 9 ✓ Step 4 — recover the chosen subset S = { item 1, item 2 } = { (6,4), (8,5) } weight = 9 ≤ 10, true value = 14 Step 5 — compare with the true optimum Brute-force OPT on this small input = 14 (same subset) FPTAS / OPT = 14 / 14 = 100 % ≥ (1 − ε) = 80 % ✓
